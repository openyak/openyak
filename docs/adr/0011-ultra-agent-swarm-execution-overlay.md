# Ultra is a model-directed Agent Swarm execution overlay

Ultra is an execution overlay independent of the Plan / Ask / Auto permission mode. Plan, Ask, and Auto continue to decide what the Agent may do; `standard` versus `ultra` decides whether the coordinating model may use the `swarm` Tool. Ultra therefore does not silently relax permissions, choose a different Provider, or replace the parent Agent.

The **SwarmCoordinator Module** exposes one narrow fork/join **Interface**: run a bounded list of child assignments in the parent runtime context and return their ordered outcomes. The `SwarmTool` is the model-facing **Adapter** over that Interface. Its **Implementation** creates 2–4 durable child Sessions, one AgentRun per assignment, then joins their persisted results so the parent Agent can synthesize and verify the final answer. This model-directed design replaces the old manual TaskBatch UI/form and task-batch API; users express the goal in the normal composer and the model decides whether and how to delegate.

Every child inherits the parent Workspace, Project, Provider, model, reasoning setting, and effective PermissionRules. Child Sessions always run in standard execution mode, which bounds recursion by configured Agent depth. A child GenerationJob observes parent cancellation through a one-way linked abort signal, so a child timeout cannot cancel its parent or siblings. Interactive permission, question, and plan-review requests are proxied through the parent response broker with child context. The parent Session remains the sole interaction owner.

The default `research` Agent is strictly read-only and its independent AgentRuns execute concurrently, bounded by the configured concurrency limit. Tools are treated as mutation-capable unless they explicitly declare the existing concurrency-safe/read-only contract; MCP Tools must set `readOnlyHint`. Any Agent with an effective mutation-capable Tool acquires the per-Workspace mutation lease, so such AgentRuns serialize even when different Swarms overlap. That scheduling **Seam** preserves Workspace consistency without leaking concurrency rules into orchestration prompts.

Each Swarm owns exactly one persisted `SwarmPart` in the parent assistant Message. It is updated in place as a complete snapshot with `schema_version: 1` and a monotonically increasing `revision`; every `swarm-state` SSE event carries the same full shape. State is committed before publication, allowing replay, de-duplication, and database refetch to converge on the same revision. This single protocol has high **Leverage** and **Locality**: persistence, streaming, and UI rendering share one lifecycle contract instead of reconstructing state from child event buffers.

The parent-scoped Subagents UI adopts only the workflow-relevant parts of the Codex pattern: Active/Done lifecycle sections, live status, concise output previews, child transcripts, and return navigation. Decorative per-agent identity such as generated avatars or avatar color systems is an explicit non-goal; task title, status, timestamp, and output are the durable identity users need.

Child failures are isolated. Remaining AgentRuns continue, the join returns `partial` when at least one child succeeds, and the parent receives both successful output and failure metadata for synthesis. Child Jobs are registered with the StreamManager while active, so aborting or deleting any Session reaches its generation. Deleting a parent Session discovers and deletes its full descendant Session tree leaf-first, preventing orphan AgentRuns and stale indexes. In addition to the 2–4 member per-call bound, one parent generation has a hard total child-Agent reservation budget; changing Tool arguments cannot bypass it.

This architecture adds concurrency and token cost, but keeps the conceptual **Depth** in one coordinator Module. The parent loop, Provider Interface, Tool executor, and Session storage remain unchanged consumers of a normal Tool result and a normal Part.
